# ADR-004: Secrets Management — Spring Cloud AWS Secrets Manager

**Status:** Accepted  
**Date:** 2026-06-19

## Context

ECS tasks need runtime access to sensitive configuration: the RDS password generated by Terraform, the OpenAI API key, and optionally the Anthropic API key. These must never appear as plaintext in task definitions, environment variables, or version-controlled files.

Three credible options exist:

| Option | How | App awareness | Startup dependency on SM |
|---|---|---|---|
| ECS-native task def `secrets` | ECS agent fetches from SM, injects as env vars | None — app sees plain env vars | No (agent fetches once; SM downtime after task start is irrelevant) |
| Spring Cloud AWS SM | App calls SM at startup via `spring.config.import` | Explicit — app fetches its own secrets | Yes (SM must be reachable at startup) |
| SSM Parameter Store (SecureString) | App or ECS agent reads SSM | Either | Either |

## Decision

Use **Spring Cloud AWS Secrets Manager** (`io.awspring.cloud:spring-cloud-aws-starter-secrets-manager`, version 3.2.1).

Configuration via `spring.config.import: aws-secretsmanager:/knowledge-assistant/{env}` in the profile-specific yml. The SM secret stores a JSON object whose keys (`DB_PASSWORD`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`) match the `${...}` property expressions already present in `application.yml`. Spring Cloud AWS injects them as a property source — no changes to `application.yml` are required.

**Local development:** `spring.config.import: optional:aws-secretsmanager:/knowledge-assistant/local` with `spring.cloud.aws.secretsmanager.endpoint: http://localhost:4566`. The `optional:` prefix means startup succeeds if LocalStack is not running; the app falls back to `application-local.yml` defaults. LocalStack's `02-create-secrets.sh` seeds the secret with dev-safe placeholder values at startup.

**Production:** No `optional:` prefix. A task that cannot read its secrets must not start. The ECS health check will mark the task unhealthy and the ALB will not route traffic to it.

## Why Not ECS-native injection

ECS-native injection is operationally simpler (no library dependency) and makes the app resilient to SM downtime after the task has started. For a portfolio project, however, Spring Cloud AWS demonstrates a broader integration pattern and makes the configuration loading visible in application code rather than invisible infrastructure behaviour.

For a regulated production deployment where SM downtime tolerance matters, switching to ECS-native injection requires: removing the `spring.config.import` lines, removing the Spring Cloud AWS dependency, and adding `secrets` blocks to the ECS task definition Terraform config. No business logic changes are needed — the app still reads `${DB_PASSWORD}` regardless of which mechanism populates that property.

## Why Not SSM Parameter Store

SSM Parameter Store SecureString is cheaper ($0.05 per 10,000 API calls vs $0.40/secret/month for SM). However, SM is the correct tool for secrets: it provides automatic rotation, versioning, cross-account sharing, and a dedicated audit trail in CloudTrail. SSM is better suited for non-secret configuration (feature flags, service endpoints). The cost difference is negligible for this project.

## Consequences

**Good:**
- SM unavailability fails the deployment loudly (startup exception, task never becomes healthy)
- Secret JSON keys are self-documenting alongside the Spring property names they supply
- Local dev works against LocalStack with the same code path as production, just a different endpoint
- Rotating a secret requires updating the SM secret version and force-restarting the ECS service — no Terraform changes needed

**Bad:**
- App startup is slower (SM API call is on the critical path)
- SM must be reachable from ECS tasks (NAT Gateway egress or VPC endpoint required)
- Spring Cloud AWS BOM must be kept compatible with Spring Boot version upgrades
