# Phase 8 — AWS Deployment

## Goal

Production-ready infrastructure: ECS Fargate, RDS PostgreSQL with pgvector, S3, Secrets Manager, ALB, and a CI/CD pipeline. The application is containerised, secrets are never in plaintext, and every push to `main` deploys automatically.

---

## Scope Split — Locally Testable vs. Apply on AWS Account Creation

Phase 8 has no AWS account yet, so deliverables are split into two layers:

**Layer A — Locally runnable now:**
- `docker/prod/Dockerfile` — multi-stage build; testable with `docker build` + `docker run`
- Spring Cloud AWS Secrets Manager integration — LocalStack already runs `secretsmanager`; `application-local.yml` points at `http://localhost:4566`. The app resolves secrets from LocalStack at startup.
- `docker/local/localstack-init/02-create-secrets.sh` — seeds the `/knowledge-assistant/local` SM secret at LocalStack boot.

**Layer B — Code-complete, apply when AWS account exists:**
- 9 Terraform modules (vpc, security, ecr, rds, s3, secrets, iam, alb, ecs)
- `environments/prod/` wires them
- `bootstrap/` creates the S3 + DynamoDB Terraform state backend (run once)
- `.github/workflows/cd.yml` — GitHub Actions deploy pipeline

## What Phase 8 Delivers

- **Production Dockerfile** — multi-stage, non-root, JRE Alpine, `MaxRAMPercentage` JVM tuning
- **Spring Cloud AWS Secrets Manager** — `spring.config.import: aws-secretsmanager:/knowledge-assistant/prod` in the `prod` Spring profile; `optional:aws-secretsmanager:/knowledge-assistant/local` in the `local` profile pointing at LocalStack
- **`application-prod.yml`** — activated by `SPRING_PROFILES_ACTIVE=prod` (set in ECS task def); disables Swagger UI; imports secrets from SM
- **LocalStack SM seed script** — `02-create-secrets.sh` creates `/knowledge-assistant/local` with dev-safe placeholder values at LocalStack startup
- **Terraform VPC module** — VPC, 2 public subnets (ALB), 2 private subnets (ECS + RDS), Internet Gateway, single NAT Gateway
- **Terraform security module** — ALB SG (80/443 open), ECS SG (8080 from ALB only), RDS SG (5432 from ECS only)
- **Terraform ECR module** — image registry with scan-on-push and lifecycle policy (keep last 10 images)
- **Terraform RDS module** — PostgreSQL 16, `db.t3.micro`, encrypted at rest, random password generated by Terraform, Performance Insights enabled
- **Terraform S3 module** — document bucket, SSE, versioning, non-current version expiry
- **Terraform secrets module** — SM secret at `/${project}/${env}` with DB password + API keys
- **Terraform IAM module** — task execution role (ECR pull + log write + SM read at startup); task role (S3 R/W + SM read at runtime)
- **Terraform ALB module** — ALB, target group (HTTP 8080, health check `/actuator/health`), optional HTTPS (`enable_https` flag, ACM certificate, 80→443 redirect)
- **Terraform ECS module** — ECS cluster (Container Insights), task definition (0.5 vCPU / 1 GB), ECS service (min 100% / max 200% deployment), CPU autoscaling 1→4 tasks at 70%
- **Bootstrap module** — S3 bucket + DynamoDB table for Terraform remote state (run once)
- **`infrastructure/README.md`** — deploy guide, OIDC setup, cost estimate, teardown instructions
- **`.github/workflows/cd.yml`** — OIDC auth → `docker build` → ECR push (SHA + latest tags) → `ecs update-service --force-new-deployment` → `ecs wait services-stable`
- **`ci.yml` cleanup** — Phase 8 placeholder comment removed; `docker-build` job now validates the real production Dockerfile

## Architecture Decisions

### Spring Cloud AWS SM over ECS-native task def secrets injection

ECS supports injecting SM secrets directly into task environment variables via the `secrets` block in the task definition. The app would receive `DB_PASSWORD` as a plain env var and would be completely unaware of SM. This is simpler and makes the app resilient to SM downtime after startup.

Spring Cloud AWS Secrets Manager (`spring.config.import: aws-secretsmanager:/...`) is more explicit: the app fetches the SM secret at startup, and the JSON keys become Spring property sources that override `${...}` defaults in `application.yml`. This means SM unavailability at startup fails the deployment, which is the correct behaviour — a task that can't reach its config store shouldn't start serving traffic.

The downside worth naming: Spring Cloud AWS adds a library dependency for something AWS can do natively at no cost. The tradeoff is justified here by the portfolio signal it gives and the explicit observability of startup failures.

The `optional:` prefix in the local profile (`optional:aws-secretsmanager:/knowledge-assistant/local`) prevents startup failure when LocalStack isn't running — the app falls back to `application-local.yml` defaults. No `optional:` in the prod profile: a prod task that can't reach SM must fail hard.

### SM secret keys match existing `${...}` expressions — no application.yml changes

The SM secret stores `{"DB_PASSWORD": "...", "OPENAI_API_KEY": "..."}`. Spring Cloud AWS injects these as Spring properties named exactly `DB_PASSWORD`, `OPENAI_API_KEY`, etc. The existing `${DB_PASSWORD:ka_password}` expressions in `application.yml` resolve to the SM values — no other changes to application config needed.

The alternative (storing keys as `spring.datasource.password`, `spring.ai.openai.api-key`) is more idiomatic Spring but would require changing `application.yml` to remove the env var wrapper — unnecessary churn.

### Dockerfile: multi-stage Maven → JRE Alpine

Stage 1 (`maven:3.9-eclipse-temurin-21-alpine`) builds the fat JAR. Stage 2 (`eclipse-temurin:21-jre-alpine`) is the runtime. The JRE image is ~200MB vs ~400MB for the JDK image.

The dep-caching layer trick: `COPY backend/pom.xml .` → `RUN mvn dependency:go-offline` before copying source. The dependency layer is only invalidated when `pom.xml` changes, not on every source edit. This halves CI rebuild time after the first build.

Non-root user (`addgroup -S app && adduser -S app -G app`) is a container security requirement: if the app is compromised, the attacker doesn't have root inside the container.

### JVM flags for container-aware memory management

```
-XX:+UseContainerSupport      — JVM honours cgroup memory limits (default in Java 11+; explicit here for clarity)
-XX:MaxRAMPercentage=75.0     — 75% of container memory for heap. For 1GB ECS task: ~768MB heap.
                                The remaining 25% covers metaspace, thread stacks, and off-heap.
-Djava.security.egd=file:/dev/./urandom — avoids startup blocking on /dev/random entropy
```

`-Xmx` is not used. A fixed heap size becomes wrong if the task's memory is later resized via Terraform. `MaxRAMPercentage` scales automatically with container memory.

### Single NAT Gateway

Two NAT Gateways (one per AZ) costs ~$64/month for a portfolio project. The cross-AZ data transfer cost of a single NAT is negligible at this scale. Set to one per AZ if you're paying for multi-AZ RDS and need true zone independence.

### `lifecycle { ignore_changes = [task_definition] }` in ECS service

Without this, `terraform apply` would always reset the task definition to whatever `var.image_tag` is (default: `latest`). The CD pipeline updates the running service by calling `ecs update-service --force-new-deployment` after pushing a new image — Terraform should not interfere with that. This lifecycle block means: Terraform manages the ECS service configuration (desired count, networking, load balancer wiring), but CI/CD manages what image is running.

### RDS pgvector via Flyway — no custom parameter group needed

RDS for PostgreSQL 16 ships with pgvector as an available extension. No custom parameter group or `shared_preload_libraries` setting is required. Flyway's `V1__init.sql` already contains `CREATE EXTENSION IF NOT EXISTS vector;` and runs at first app boot. The extension is enabled in the database by the app itself — Terraform only needs to create the RDS instance.

### OIDC for GitHub Actions — no long-lived IAM keys

The CD workflow uses `aws-actions/configure-aws-credentials@v4` with `role-to-assume`. GitHub Actions mints a short-lived OIDC token per job; AWS exchanges it for temporary STS credentials scoped to the deploy role. No IAM access key or secret is stored in GitHub Secrets. The deploy role grants only ECR push and ECS `update-service` — nothing broader.

### ALB HTTPS optional (`enable_https = false` by default)

ACM certificates require DNS validation on a domain you control. Without a Route53 hosted zone, HTTPS cannot be provisioned. The `enable_https` variable defaults to `false`, which creates an HTTP-only ALB listener. Set `enable_https = true` + `domain_name = "your.domain.com"` when you have a domain. The `alb_validation_options` output provides the DNS CNAME records needed to complete ACM validation.

### SM `recovery_window_in_days = 0`

By default, Secrets Manager holds deleted secrets for 30 days (at $0.40/secret/month during that period). `recovery_window_in_days = 0` enables immediate deletion, which matters for portfolio teardown. Set to `30` for a real production secret where recovery is needed.

## Sequence: CD Pipeline

```
Developer pushes to main
  │
  ├─ GitHub Actions: cd.yml triggered
  │
  ├─ aws-actions/configure-aws-credentials@v4 (OIDC token exchange → STS credentials)
  │
  ├─ aws-actions/amazon-ecr-login@v2
  │
  ├─ docker build -f docker/prod/Dockerfile . → knowledge-assistant:{sha} + :latest
  │
  ├─ docker push ECR registry (two tags)
  │
  ├─ aws ecs update-service --force-new-deployment
  │     ECS pulls the new :latest image
  │     New task starts → SM secret fetched (Spring Cloud AWS) → Flyway runs → health check passes
  │     ALB drains old task (deployment_minimum_healthy_percent = 100 means new task starts before old stops)
  │
  └─ aws ecs wait services-stable
```

## Known Limitations (deferred)

- **HTTPS:** Disabled by default (`enable_https = false`). Enable when a Route53 domain is available.
- **Single NAT Gateway:** Cross-AZ traffic if ECS tasks scale into the second AZ. Acceptable for portfolio.
- **No staging environment:** Only `prod`. Add a `staging/` environment under `environments/` if needed.
- **Token usage not populated in chat messages:** `promptTokens`, `completionTokens`, `modelUsed` on `ChatMessage` remain `null` until Phase 9 (Observability) adds richer `LlmPort` response types.
- **No auth:** JWT authentication is Phase 10. All endpoints are currently open.
